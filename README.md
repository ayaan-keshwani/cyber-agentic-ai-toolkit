# Cybersecurity Agent Toolkit for SMBs

Agentic AI toolkit for small and medium business owners to improve cybersecurity hygiene and get immediate, vetted guidance during incidents. Uses **Gemini 2.5 Flash** and the **Google Agent Development Kit (ADK)**. No Vertex AI or external databases: guidance is stored in YAML/JSON, session state for the agents in a local SQLite file, and the web UI stores chat history in the browser.

## Features

### Web application (recommended)

- **React + FastAPI**: The app serves a single-page UI from `frontend/dist` and JSON/streaming APIs under `/api`. Run `python run_web.py` after building the frontend (see [Setup](#setup)).
- **Chat threads**: Multiple chats in a sidebar; **threads persist in the browser** (`localStorage`) across reloads. Each thread is tied to an **Email Protection** or **Incident Response** agent for its lifetime (switch agent only in an empty thread).
- **Streaming replies**: Chat responses stream in over NDJSON.
- **Business profile**: Editable from the UI (name, business, email stack, IT support, cyber insurance, policy text). **Insurance declarations** can be uploaded as PDF or TXT; the backend extracts text, optionally parses **inclusions and exclusions** (Gemini-assisted when `GOOGLE_API_KEY` is set, with a fallback splitter), and stores them on the profile.
- **Reset**: A full reset clears the default business profile, SQLite sessions, and (from the profile flow) client-side chat storage where wired.

### Email Protection agent

- **Onboarding** (one question at a time): user name, **business name**, business type, email platform (Google or Microsoft 365), whether they send sensitive files by email, IT support (**in-house**, **outsourced**, or **none**), and whether they have **cybersecurity insurance**.
- If they have insurance, they are prompted to **upload** a declarations page (chat **Upload** button or profile); after upload, **inclusions** and **exclusions** are shown as bullet lists (monetary lines first), then a `---` separator, then the **email hardening checklist** with the post-insurance transition text.
- **Checklists** cover passwords, MFA, connected apps, Drive/OneDrive sharing, and phishing awareness, with stack-specific YAML under `data/guidance/checklists/`.
- **Playbooks** for suspicious email, phishing (with password-reset branch), unusual login, and BEC—only from approved scenario YAML.
- **Serious cyber incidents** (ransomware, encryption, data breach, cyber extortion, etc.) are **not** handled here: the user is told to start a **new chat** with the **Incident Response** agent so incident guidance stays in the right agent.

### Incident Response agent

- **Scenarios in guidance** include ransomware, wire fraud, cyber extortion, and data breach (`data/guidance/scenarios/incident/`), plus a general fallback. The model is instructed to use **only** approved steps when it runs YAML playbooks (for example when the business **does not** have cyber insurance).
- **When the profile has cyber insurance** and the user describes a **serious incident**, the primary flow is **not** the raw YAML playbook. Instead the agent uses the **business profile** (`it_support`, `policy_inclusions`, `policy_exclusions`), reminds the user **not to pay ransoms**, and:
  - **With in-house or outsourced IT**: Tells them to involve **IT first**; collects details **one question per message** (carrier already contacted, timeline, scope, ransom/extortion if relevant, actions taken; for **data breaches**, what has been done so far / what else has been done); asks a final **additional notes** question; then produces **one** structured draft **to IT** for the insurer’s benefit—not FBI/CISA templates for the end user.
  - **With no IT**: Similar collection, then **one** draft **to the cyber insurance carrier** (not law-enforcement email templates as the main deliverable).
- If the user **already contacted the cyber insurance carrier** about the incident, the agent **does not** continue with drafts or parallel plans: it tells them to **follow the carrier’s guidance**, that this assistant **cannot add further guidance**, and to follow the **carrier’s next steps**.
- If the issue is **only** suspicious email or phishing, the user is directed to the **Email Protection** agent.

### Shared design

- **Pre-approved guidance only**: Recommendations come from curated scenarios and checklists (FTC, CISA, etc.); agents must not invent remediation steps (no ad-hoc firewall or forensic instructions).
- **Persistent ADK sessions**: SQLite-backed sessions per `(business_id, agent_type, thread_id)` so server-side conversation state survives restarts. The **same** `business_id` shares one business profile across both agents.
- **Multi-turn flow**: Follow-ups are supported; playbooks advance when applicable.

## Setup

1. **Python 3.10+** and a virtual environment:

   ```bash
   python -m venv .venv
   .venv\Scripts\activate   # Windows
   # source .venv/bin/activate  # macOS/Linux
   ```

2. **Install Python dependencies**:

   ```bash
   pip install -r requirements.txt
   ```

3. **Environment**: Copy `.env.example` to `.env` and set your **own** Google AI Studio API key (for Gemini 2.5 Flash). **Each person needs their own key**—the repo does not include one.

   ```bash
   copy .env.example .env
   # Edit .env: set GOOGLE_API_KEY=your_key
   ```

   Get a free key at [Google AI Studio](https://aistudio.google.com/apikey). Keep `GOOGLE_GENAI_USE_VERTEXAI=FALSE`. Do not commit `.env` (it is in `.gitignore`).

4. **Web UI (optional but recommended)**: Install **Node.js** (LTS), then from `frontend/`:

   ```bash
   cd frontend
   npm install
   npm run build
   ```

   The FastAPI app serves static files from `frontend/dist`. If `dist` is missing, `run_web.py` enables reload so you can develop against the API without a build.


## Running the web app

From the project root (with `.venv` activated and `GOOGLE_API_KEY` set):

```bash
python run_web.py
```

Defaults: bind `127.0.0.1`, port **8000**, opens the browser unless `OPEN_BROWSER=0`. Optional environment variables: `WEB_HOST`, `WEB_PORT`, `DEV_RELOAD=1` (reload on code changes), `OPEN_BROWSER`.

API routes include `/api/health`, `/api/onboarding`, `/api/chat/stream` (POST, NDJSON), `/api/profile` (GET/PUT), insurance declarations upload/delete/download, and `/api/reset` (full data reset for the default business id).

## Running the CLI

```bash
python run_cli.py
```

Choose **1** = Email Protection, **2** = Incident Support. First-time users complete onboarding in the CLI before normal use. Session and profile behavior matches the shared `src/app.py` logic.

**Reset memory (clean slate)** (profile, SQLite sessions):

```bash
python run_cli.py --reset
```

Then run `python run_cli.py` again without `--reset`. The web UI has a separate reset path tied to `/api/reset` and profile UI.

## Project layout

- `data/guidance/` – YAML for source documents, scenarios (**email/**, **incident/**), and checklists.
- `data/business_profiles/` – JSON per business (runtime; typically not committed).
- `data/sessions.db` – SQLite for ADK sessions (runtime; not committed).
- `data/uploads/` – Stored insurance declaration files when uploaded (layout depends on config).
- `frontend/` – React (Vite) app; production build output in `frontend/dist`.
- `src/api/` – FastAPI application (`main.py`).
- `src/agents/` – Email Protection and Incident Support ADK agents.
- `src/tools/` – ADK tools: profile, scenarios, checklists, playbook state.
- `src/storage/` – Load guidance, file-backed business profiles.
- `src/prompts/` – System instructions (`email_instruction.txt`, `incident_instruction.txt`).
- `src/insurance_parse.py` – Optional Gemini JSON parse of policy text into inclusion/exclusion bullets.
- `src/app.py` – Runners, sessions, `run_agent` / `run_agent_stream`, context injection for agents.
- `run_web.py` – Uvicorn entry for the web stack.
- `run_cli.py` – Interactive CLI entry.

## Updating guidance

- **Scenarios**: Edit YAML under `data/guidance/scenarios/` (`email/`, `incident/`, `fallback.yaml`). Each scenario has an `id`, `steps`, and optional `trigger_signals` and `applies_to`.
- **Checklists**: Edit YAML under `data/guidance/checklists/`.
- **Source documents**: `data/guidance/source_documents.yaml` for attribution.

Restart the app after YAML changes (guidance is cached in process).

## Liability and constraints

- Agents are instructed to recommend **only** actions that appear in the provided scenarios and checklists when giving step-by-step guidance, and to follow the stricter **prompt** rules for incident handling when cyber insurance and profile fields apply.
- Fallback and insurer/authority wording differ by agent and context; this toolkit is for **guidance only** and does not replace legal, insurance, or professional incident-response advice.
